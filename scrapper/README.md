# Scrapper-Service

Сервис позволяющий отслеживать изменения внесенный в репозиторий github или тему на stackoverflow.

## Использование

Проект запускается файлом main.py из директорий src

```
python main.py
```

### docker

Для запуска всего проекта через докер:

```
project/
    bot/
    srcapper/
        docker-compose.yml
```

```
docker-compose -f docker-compose.yml up --build
```

## ВАЖНО

Не забудьте создать файл .env в src/secrets/.env, пример заполнения можно посмотреть в той же директории.
src/secrets/.env_example

### Классы

```
FastAPI endpoints -> LinkService -> RawRepository/OrmRepository

Scheduler -> Clients -> Notifier
```

Классы связаны между собой абстракциями, в которые вынесен необходимый функционал. Не зависят от реализации друг друга.

### Работа с базой данный

В этой итерации была добавлена возможность работы с базой данных PostgreSQL. Работа с базой данных осуществленная в классах RawRepository, OrmReposotiry. Выбрать режим работы можно в файле src/secrets/.env.

В задании релизована пагинация на получение ссылок из бд.

По часто используемым атрибутам в бд построенны индексы.

## Функциональные требования

- [x] Приложение должно использовать базу данных для сохранения информации (пользователи, ссылки, подписки, теги) между перезапусками.

- [x] Приложение должно поддерживать CRUD-операции для работы с данными. (Поддреживаются необходимые операции)

## Нефункциональные требования

- [x] Для хранения данных используйте PostgreSQL .

- [x] Запуск Postgres осуществляется compose файл

- [x] Миграции должны быть написаны на языке SQL .

- [x] Для миграций используется Alembic

- [x] Схема БД должна находиться в каталоге migrations/ в корне проекта.

- [x] Запуск миграций должен выполняться программно из кода приложения.

- [x] Реализуйте 2 способа работы с БД : низкоуровневый доступ (raw SQL) и высокоуровневый доступ (ORM/Query Builder).

- [x] Выбор способа работы с БД осуществляется через конфигурацию: access-type=SQL или access-type=ORM.

- [x] Операции, затрагивающие несколько таблиц (например, создание подписки), должны выполняться в рамках транзакции .

## Тестирование

Написаны unit тесты на все основные функции, все вызовы настоящих api замоканы в фикстурах.

Для запуска выполните `pytest tests`

### TestContainers

Реализованно Бонусное задание TestContainers. Для его проверки поместите файлы в директорию следующим образом:

```
project/
    bot/
    srcapper/
        testscontainers
```

В директорий testscontainers выполните `pytest`

Будут подняты контейнеры с сервисами и протестироавнно http соединение.
